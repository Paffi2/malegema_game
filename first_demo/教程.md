**在 Godot 里的具体复现流程（一步步做）**

1. 搭骨架：先做最小可玩局

- 用你现有 Godot 项目起步：project.godot。
- 先做一个 Board 场景，能显示 tiles，支持点击两张牌。
- 先不做商店、不做奖励、不做特殊牌，只做基础匹配与消除

2. 复现数据定义

- 把牌定义从 game.ts 迁移成 Godot 数据（建议字典或 Resource）。
- 必须保留 cardId、suit、rank、colors、points。
- 材质系统也要保留：bone/topaz/sapphire/garnet/ruby/jade/emerald/quartz/obsidian。

3. 复现地图和坐标系统

- 直接迁移 responsive.ts 的三维布局数据。
- 坐标语义保持一致：x/y/z。
- 复现 mapGet、overlaps、fullyOverlaps、isFree 这组判定逻辑（都在 game.ts）。

4. 复现可解发牌算法

- 完整照搬 setupTiles.ts 的思路：先用 dummy 牌反复“可消对抽取”，确保布局可解，再把真实牌对反向填回。
- 这是 whatajong 手感的关键，不建议自己改算法。

5. 复现选牌与结算主循环

- 对照 game.ts 的 selectTile 流程：
  1) 首次点击选中
  2) 二次点击同一张取消
  3) 两张匹配则删除并算分
  4) 两张不匹配尝试跳跃类特殊逻辑
  5) 每次操作后检查 gameOverCondition
- 这里先只做 cardsMatch、deleteTiles、getPoints、gameOverCondition。

6. 逐个接入特殊牌模块

- 顺序建议：Dragons → Phoenixes → Gems → Mutations → Winds → Jokers。
- 对应文件见上面的“第二优先级”。
- 每加一个模块就跑一轮固定种子局，确认没有破坏基础逻辑。

7. 复现回合系统（roguelike 外循环）

- 先迁移 round 生成与难度曲线：来自 runState.tsx。
- 再迁移商店条目生成、购买、升级合成（3 合 1 链路）。
- 阶段切换按 run.tsx 的 stage 状态机做。

8. 做持久化

- 原项目状态是持久化的（run/game/deck/tile 各自独立）。
- Godot 里建议一份 SaveData 管 run、当前 round、deck、tileDb。
- 存档触发点：进入新阶段、回合结束、购买后。

9. 最后补表现层

- 动画、音效、tutorial、视觉细节最后做。
- 你当前 Godot 里已有启动脚本可复用入口：start_menu.gd。

**复现时最容易踩坑的点**

1. 坐标判定差 1 个单位就会导致 isFree 判错，进而整个“无解/有解”结果错。
2. Joker 洗牌不是纯随机重排，是先抽可消顺序再回填配对。
3. Winds 位移要逐层处理，并按方向排序移动，否则会互相覆盖。
4. runState 的升级链路是 3 合 1，不是简单叠加属性。

**四、推荐复现顺序（最省时间）**
第 1 周：打基础

1. 学会 Godot 场景、节点、信号、输入、资源系统。
2. 先做一个 Tile 节点，支持选中、高亮、删除。

第 2 周：核心规则

1. 实现自由牌判定、配对判定、可配对数统计。
2. 实现 game over 条件：empty-board 和 no-pairs。
3. 实现基础计分与计时罚分。

第 3 周：高级规则

1. 迁移 Dragon Run / Phoenix Run / Gem / Mutation / Wind / Joker / Shadow。
2. 实现完整结算顺序（必须保持同顺序，否则手感和结果会偏）。

第 4 周：回合循环

1. 实现 run stage 流转：game -> gameOver -> reward/shop -> next round。
2. 接入商店购买、reroll、freeze、三合一材质升级。
3. 实现 seed 固定随机，保证可复现。

第 5 周：打磨

1. 动画和音效。
2. 新手引导和文案。
3. 存档与恢复。

**五、最关键的“不要踩坑”点**

1. 结算顺序不能乱。原项目是删牌后按固定链路触发特效，再判终局。
2. 随机必须可复现。Godot 里要统一种子来源，避免每次结果漂移。
3. 先做纯逻辑层。UI 只是读状态，不要把规则写进 UI 回调。
4. 小丑洗牌不是简单 random shuffle，而是“构造可解序列再回填”。
5. Shadow 会影响颜色与材质逻辑，和分数计算强耦合，单测必须补齐。
